Project Idea

I aim to develop a 2D, tile-based game where the character can move around and discover things in a fantasy world. It is inspired from titles such as OneShot and Undertale, which tile-based as well. These indie games did not rely on great graphics or complicated game mechanics, but rather excellent story-telling and an element of "aliveness" that is achieved through storing player decisions regardless of save files. I intend to have puzzles the player can solve and will work towards creating a fundamental story line.

The digital game market is becoming increasingly bigger as games are constantly being pushed to its limit along with the exponential growth of computer processors and graphics cards. However, in this rush to push out as many games as possible, as flashy and beautiful as possible, the market has become oversaturated with similar concepts such as FPS or battle royale. In this day and age, it is not often that we see a game orientated around the story rather than the graphics or game. That is why I feel like we should have more games with an indie style, nostalgic 16-bit game with good atmosphere.

I imagine that we can begin this project with making the main character sprite that the player will control. Then we can create a 2D tiling system and import some basic ground textures to test whether or not the game can render properly. Most of the terrain textures will have to be seamless in order to create the illusion that the ground is one whole texture, rather than a repeating pattern of the same tile. Next we implement a camera that will follow our player wherever it moves, as well as code the function for players to move on an x-y coordinate axis. We have to make sure the velocity of the player movement is not too fast or slow, and then finish bu binding some keys to the movement. Next, I plan to create a series of walking poses for my main sprite in order to animate the player character so it seems more lively. We can accomplish this by running through a loop of images depending on which way the character is moving. After this, I would like to work on some terrain and immovable objects, starting with walls and working my way up to more complicated terrain. Although this is a 2D game, we have to keep in mind that we are not working with a top-down view, but rather a side-view, so there will be some 3D elements that we will need to implement. For example, if the player walks behind a tree or wall, the game should know not to render the character; and if the player is in water, we can render half of the player and add a bobbing animation with water splashing to smooth out the cut of the sprite in the water. Once we have a general terrain completed, we can begin to work on interactable items and maybe an inventory. We could add item descriptions to give a bit of depth to the game. Then we can begin working on NPC's with randomly generated movement and dialogue options. By this point, we have created all the fundamental game mechanics and can begin writing a storyline and implementing different rooms for characters to roam and other rooms for puzzles. I want to create a similar game mechanic in OneShot, where items that are found can be used on puzzles and combined with other items to solve especially difficult ones. We can make the game feel more alive by adding dialogue for object interactions as well. The premise of the story will be that our main character wakes up in a trapped room of some sort and has no clue of how he got there. It feels like an overused trope but I think that it is a classic element of RPG adventure games that just make the transition from a black screen to gameplay more seamless. I think it is also a good way to let the user experience the game mechanics immediately, rather than going through tutorials.

There are many platforms with which 2D games can be made, such as Unity, GameMaker Studio, and Eclipse. Each of these programs utilize different programming languages and every one has a varying level of difficulty. Although it would depend on the skills of our group members, I would either choose between Unity or GameMaker Studio. Both of these platforms have a wide range of tutorials we can use as reference, and both have their own advantages such as scripting API and online forums that provide solutions. GameMaker studio is more of a drag-and-drop environment while Unity relies a bit more on Javascript and C#. The graphics can be made on paint or Photoshop, as should only be 16-bit and not terribly difficult to design. We can also use royalty-free pixel graphics that are available online, which will allow us to focus more of our efforts on gameplay and mechanics.

The skills required for this project would depend the experience of our members, but assuming that everyone has a fundamental level of game design and math, its should be feasible to complete most of the fundamental game mechanics. It would be preferable if group members have basic knowledge of C#, which will be used for some of the core components of the game such as movement, collisions, and interactions.

In the circumstance that this project is successfully completed, we can release it to platforms such as Steam for players to enjoy. We, as developers, can also take this opportunity to receive criticism which will help us grow as game designers and increase our repertoire.
